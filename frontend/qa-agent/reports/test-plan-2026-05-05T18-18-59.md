# Plan de Pruebas: Plataforma Marketplace "ReUse" (ITESO)

## 1. Introducción
El presente documento detalla la estrategia de aseguramiento de calidad (QA) para las nuevas funcionalidades del marketplace "ReUse". El alcance se centra en tres pilares fundamentales: la publicación de artículos, la interacción pública mediante comentarios y el sistema de reacciones sociales. El objetivo es garantizar la integridad, seguridad y usabilidad del sistema, cumpliendo con los estándares de la comunidad ITESO.

---

## 2. Estrategia de Pruebas
Se adoptará un enfoque de **Automatización de Pruebas de Extremo a Extremo (E2E)** utilizando **Playwright** con **TypeScript**. 

*   **Entorno:** Las pruebas se ejecutarán en un entorno de *Staging* espejo de producción.
*   **Gestión de Datos:** Se utilizará un conjunto de datos controlado en PostgreSQL. Los estados del sistema se resetearán mediante scripts de limpieza antes de cada ejecución.
*   **Automatización con IA:** Se empleará un agente de IA para la generación de datos de prueba sintéticos (ej. comentarios aleatorios, descripciones largas) y para la validación de la interfaz responsiva, asegurando que los elementos dinámicos (Optimistic UI) se comporten según lo esperado.
*   **Pipeline:** Integración continua mediante GitHub Actions para ejecución automática ante cada *Pull Request*.

## 4. Pruebas Negativas y de Borde

*   **Validación de Entradas:**
    *   Inyectar caracteres especiales o scripts (XSS) en formularios de comentario/publicación.
    *   Ingresar descripciones > 500 caracteres (comentarios) y verificar manejo del error.
*   **Seguridad:**
    *   Acceder a los endpoints POST/DELETE mediante herramientas como Postman/Curl sin token de autenticación (Esperado: 401 Unauthorized).
    *   Intentar manipular el `product_id` en peticiones DELETE de otros usuarios (Esperado: 403 Forbidden).
*   **Condiciones de Borde:**
    *   Simular latencia de red elevada para verificar el estado de "Loading" en componentes.
    *   Prueba de concurrencia: Dos usuarios reaccionan al mismo ítem simultáneamente para asegurar la integridad de los contadores en base de datos.

## 5. Criterios de Aceptación

Para considerar que estas funcionalidades han superado el proceso de QA, deben cumplirse los siguientes criterios:

1.  **Cobertura del Código:** >85% de cobertura en pruebas unitarias de servicios (Backend) y >70% en pruebas E2E (Frontend).
2.  **Performance:** El tiempo de carga de las secciones de comentarios y reacciones debe ser inferior a 1.5s bajo condiciones normales de red.
3.  **Cross-Browser:** Funcionalidad validada en Chrome, Firefox y Safari (versiones actuales).
4.  **Responsive Design:** Ningún componente de la UI debe sufrir desbordamiento (overflow) en resoluciones móviles (desde 320px).
5.  **Cero Bugs Críticos:** No deben existir defectos abiertos de severidad "Bloqueante" o "Alta" previos al paso a producción.
