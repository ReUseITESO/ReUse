# Plan de Pruebas: Plataforma Marketplace "ReUse" (ITESO)

**Preparado por:** QA Automation Lead  
**Fecha:** 22 de mayo de 2024  
**Proyecto:** ReUse - Marketplace Circular ITESO  

---

## 1. INTRODUCCIÓN
El presente documento detalla el plan de pruebas para la plataforma "ReUse". El alcance se centra en tres funcionalidades críticas del ecosistema: la creación de publicaciones, la gestión de comentarios públicos y el sistema de reacciones (likes/dislikes). El objetivo es asegurar la integridad de los datos, la correcta aplicación de reglas de negocio y una experiencia de usuario fluida bajo el stack tecnológico (Next.js, TypeScript, PostgreSQL).

---

## 2. ESTRATEGIA DE PRUEBAS
Para garantizar una alta calidad y velocidad en el ciclo de entrega, se adoptará la siguiente estrategia:

*   **Enfoque Automatizado:** Uso de **Playwright** para E2E (End-to-End) por su capacidad de manejar estados asíncronos y múltiples navegadores.
*   **Integración con IA:** Utilización de agentes de IA para la generación dinámica de datos de prueba (fuzzing básico) y validación de UI mediante visión computacional para asegurar que el feedback visual (ej. estados de carga) sea correcto.
*   **Niveles de prueba:**
    *   **Unitarias:** (Vitest) para lógica de validación de formularios y cálculos de conteo.
    *   **API:** (Supertest) para validar contratos de endpoints RESTful.
    *   **E2E:** (Playwright) centradas en los flujos críticos de usuario.
*   **Gestión de Estados:** Uso de *fixtures* de Playwright para inyectar sesiones autenticadas (`storageState`) y evitar logueos manuales repetitivos.

## 3. ESCENARIOS DE PRUEBA

### HU-01: Publicación de ítems
| ID | Escenario | Pasos | Resultado Esperado |
| :--- | :--- | :--- | :--- |
| TC-01 | Publicación Exitosa | Completar formulario con datos válidos y subir imagen. | Mensaje de éxito, redirección al detalle, item visible. |
| TC-02 | Validación Campos | Intentar enviar formulario con campos obligatorios vacíos. | Validación en tiempo real (front) y error 400 (back). |

### HU-02: Comentarios públicos
| ID | Escenario | Pasos | Resultado Esperado |
| :--- | :--- | :--- | :--- |
| TC-03 | Post de Comentario | Usuario autenticado escribe y publica comentario. | Comentario aparece instantáneamente (Optimistic UI). |
| TC-04 | Borrado por Dueño | Ítem owner elimina un comentario ofensivo. | Comentario removido de la vista y BD. |
| TC-05 | Integridad de UI | Ver ítem sin comentarios. | Mostrar estado vacío: "Sé el primero en comentar". |

### HU-03: Reacciones (Like/Dislike)
| ID | Escenario | Pasos | Resultado Esperado |
| :--- | :--- | :--- | :--- |
| TC-06 | Toggle de Reacción | Clic en Like, luego clic en el mismo icono. | La reacción se activa y luego se remueve (estado nulo). |
| TC-07 | Cambio de Reacción | Clic en Like, luego clic en Dislike. | La reacción cambia (Like desaparece, Dislike se marca). |

## 4. PRUEBAS NEGATIVAS Y DE BORDE
*   **Seguridad:** Intentar realizar un `POST` a `/comments` sin token JWT o con un token expirado (esperado: 401 Unauthorized).
*   **Límites:** Ingresar 501 caracteres en el campo de texto de comentarios (esperado: Error 400, validación bloqueada).
*   **Concurrencia:** Simular dos usuarios reaccionando simultáneamente al mismo producto para verificar que los contadores no tengan condiciones de carrera (Race Conditions).
*   **Datos inválidos:** Inyectar scripts (XSS) en la descripción del ítem o comentarios (esperado: Sanitación exitosa en backend).

---

## 5. CRITERIOS DE ACEPTACIÓN
El software se considerará listo para despliegue si cumple con:

1.  **Cobertura de código:** Mínimo 80% en los módulos de Marketplace.
2.  **Ejecución E2E:** 100% de los escenarios críticos (Happy Paths) superados en entorno de Staging.
3.  **Performance:** El tiempo de carga de la lista de comentarios debe ser inferior a 500ms bajo condiciones normales de red.
4.  **Integridad de Contratos:** Todos los endpoints responden según la especificación definida en `contracts.md`.
5.  **Accesibilidad:** Formularios navegables mediante teclado y presencia de etiquetas `aria-label` en iconos de reacción.
